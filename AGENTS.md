# AGENTS.md

## Project

Turing Pattern Lab

This project is a React + TypeScript + Vite web application for interactively exploring Turing patterns generated by reaction-diffusion systems.

The MVP should prioritize a smooth visual experience, simple controls, and maintainable implementation.

## Product Goal

Create a browser-based science-art application where users can:

- See Turing patterns generated in real time
- Switch between natural-pattern presets
- Adjust key parameters
- Randomize the simulation
- Pause and resume the animation
- Export the current pattern as PNG

## Recommended Tech Stack

Use the following stack unless there is a strong reason not to:

- React
- TypeScript
- Vite
- HTML Canvas
- CSS Modules or plain CSS
- Vitest for unit tests where appropriate

Do not introduce a backend for the MVP.
Do not introduce WebGL unless explicitly requested later.
Do not introduce heavyweight UI libraries unless clearly justified.

## Implementation Guidelines

### Simulation

- Implement the MVP using the Gray-Scott reaction-diffusion model.
- Keep simulation logic separate from React components.
- Prefer pure functions for simulation updates where practical.
- Avoid placing heavy simulation logic directly inside JSX.
- Use typed parameter objects for simulation settings.

Suggested domain model:

```ts
type ReactionDiffusionParams = {
  feed: number;
  kill: number;
  diffA: number;
  diffB: number;
};
```

### Rendering

- Render the simulation using HTML Canvas.
- Keep rendering logic separate from parameter control UI.
- The animation loop should use `requestAnimationFrame`.
- Ensure cleanup of animation frames on component unmount.
- Avoid unnecessary React re-renders during the simulation loop.

### Presets

Create at least five presets:

- Zebra
- Giraffe
- Leopard
- Coral
- Maze

Each preset should define a name, description, and simulation parameters.

### UI

The MVP UI should include:

- Canvas display area
- Preset selector
- Feed slider
- Kill slider
- Random button
- Pause/Resume button
- Save PNG button
- Short educational explanation

The UI should be responsive:

- Desktop: Canvas and controls should be easy to view together.
- Mobile: Canvas should appear above controls, controls should be stacked vertically.

### Accessibility

- Use semantic HTML where possible.
- Provide labels for sliders and buttons.
- Avoid controls that can only be understood by color.

### Styling

- Keep the visual style clean, modern, and suitable for science-art content.
- Dark UI is acceptable if the pattern remains highly visible.
- Avoid excessive decorative UI before the MVP is functional.

## Testing

Add tests where they provide clear value.

Good test targets:

- Preset definitions exist and contain valid numeric parameters.
- Random parameter generation stays within expected ranges.
- Utility functions return valid values.

Do not over-test Canvas drawing internals in the MVP.

## Development Order

1. Initialize Vite + React + TypeScript project.
2. Implement Gray-Scott simulation state and update logic.
3. Render the simulation on Canvas.
4. Add presets.
5. Add Feed/Kill sliders.
6. Add Random, Pause/Resume, and Save PNG controls.
7. Add responsive layout.
8. Add short educational text.
9. Add minimal tests.
10. Update README if setup or usage changes.

## Quality Bar

Before considering the MVP complete:

- `npm install` succeeds.
- `npm run dev` starts the app.
- `npm run build` succeeds.
- TypeScript has no errors.
- The app works in a modern desktop browser.
- The layout is usable on a smartphone-width viewport.
- No obvious animation loop leaks or runaway re-rendering issues exist.

## Documentation Rules

- Keep `docs/requirements.md` aligned with implemented scope.
- Update `README.md` when setup, scripts, or major behavior changes.
- Prefer concise Japanese comments only where the implementation is non-obvious.
- Code identifiers should generally be English.

## Constraints

- MVP must be client-side only.
- Do not add authentication.
- Do not add database storage.
- Do not add server APIs.
- Do not add external paid services.

## Notes for Codex

When implementing, favor a small, working version over a large incomplete version.

If trade-offs are needed, prioritize this order:

1. Real-time pattern generation works.
2. Controls are responsive and understandable.
3. Code is maintainable.
4. Visual polish.
5. Advanced educational explanation.
